# 核心开发原则：绝对忠于原代码，禁止过度干预

- **绝对禁止大改和重构**：除非我明确发送了“请重写”、“请重构”或“请优化”的指令，否则**绝对不允许**对我的基础代码架构、逻辑流、命名规范进行大面积修改。
- **精准局部修复**：遇到报错或异常时，**哪里坏了修哪里**。只允许在当前报错的代码块基础上进行最小范围的局部修改，严禁因为“有更优雅的写法”而推翻重写。
- **严禁篡改 UI 与视觉设计**：完全保留现有的 CSS 样式、HTML 结构和组件布局。不得擅自更改颜色、边距、字号、动画效果等任何前端视觉表现形式。
- **尊重现有逻辑**：如果现有的代码能跑（哪怕看起来有些基础或冗余），请保留它的原貌。不要为了炫技而使用我难以理解的复杂高级语法或生僻 API。
- **代码输出规范**：在回复代码修改时，请只输出**被修改的部分**以及必要的上下文。对于未修改的部分，必须使用 `// ... 现有代码保留 ...` 的方式省略，**绝对禁止**动辄输出数百行的完整文件。
### 需求实现与交付自检 (Pre-Delivery Verification)

- **核心指令闭环核对**：在输出任何代码前，必须回顾我的原始诉求。重点核对：涉及的正则表达式、UI 美化效果、特定的逻辑 Prompt 是否**真正、完整地**实现了我的要求？严禁提供半成品或敷衍了事的代码。
- **正则与逻辑的严密性**：如果需求包含正则表达式或数据校验，请务必在内部推演边界条件。必须提供精准无误的正则，并在回复中简要说明该正则的匹配范围。如果存在潜在的漏判或误判风险，必须明确告知我。
- **美化与样式真实落地**：当我要求“美化页面”或调整 UI 时，请仔细检查 CSS 选择器的权重和 DOM 结构，确保你提供的样式代码**绝对能够生效**，而不是写出一段会被全局样式覆盖的无效代码。
- **查漏补缺与主动反馈**：在实现我的需求时，主动评估是否缺少必要的前置条件（例如：是否有未定义的变量？是否缺少某个 CSS 动画库？正则是否缺少转义？）。如果发现缺漏，**不要擅自改动我的基础架构去强行填补**，而是要在回复的开头明确向我提出：“为了实现该功能，还需要补充以下内容……”。
